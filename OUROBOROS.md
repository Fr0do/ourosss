# OUROBOROS

> The serpent eats its own tail: each project feeds the next, and the meta-process improves itself.

A research operating protocol for a solo ML researcher using Claude Code, Telegram, and remote GPUs.

---

## Pipeline

### Stage 0 · Ideation
Hypothesis + related work scan + feasibility check before any code.
Tools: `/deep-research` (literature scan + gap analysis), `gh issue create` (lock in the idea), Notion for eval framing.
Rule: no repo before a written hypothesis. An issue is the first artifact.

### Stage 1 · Bootstrap
Create LaTeX repo (`s_cot_tex` pattern), GitHub repo, register in PROJECTS.md.
Tools: `/deep-research` (scaffolds LaTeX + refs), `/feature` (files infra issues), `bot /page` (Notion project page).
Rule: repo + paper skeleton exist before the first training run.

### Stage 2 · Infrastructure
Training script, data pipeline, eval harness wired up and smoke-tested.
Tools: Sonnet subagent for implementation (>20 lines), `accelerate launch --config_file fsdp2.yaml`, ClearML init.
Rule: config + seed + commit hash logged from run zero. No reproducibility debt allowed to accumulate.

### Stage 3 · Training & Debugging
Interactive loop: launch → monitor → inspect → adjust.
Tools: `bot /run`, `bot /logs`, `bot /gpu`, `bot /disk`, `bot /ckpt`, `bot /completions`; `/check-training`, `/check-gpu`, `/check-ckpt`, `/completions` skills.
Rule: every run has a ClearML entry. Checkpoint every N steps. Kill diverging runs immediately.

### Stage 4 · Results & Analysis
Ablations, baselines, figures, tables — nothing goes in the paper without a logged run.
Tools: `bot /metrics`, `bot /eval`, `/completions` (GRPO output analysis), `/research-log` (findings log), Notion eval DB.
Rule: baselines run from the same config/env. Figures generated by script, not by hand.

### Stage 5 · Paper
Write, iterate, get feedback — parallel to late training.
Tools: `/deep-research` (related work refresh), Sonnet subagent for prose drafts, `bot /page` (Notion drafts), local `s_cot_tex`.
Rule: one source of truth — the LaTeX repo. No stale Overleaf shadow copies.

### Stage 6 · Release
Camera-ready, artifact packaging, code release, portfolio update.
Tools: `gh release create`, `/feature` (release checklist issue), `bot /sync`.
Rule: artifact + paper + code released atomically. A result without reproducible code is an anecdote.

## Operational Rules

1. **Loop closes** — every artifact (pipeline, tooling, eval) must improve the next iteration. Dead-end infra gets deleted.
2. **Issues before code** — create a GitHub issue before any feature. `fixes #N` in every commit. `/feature` automates this.
3. **Reproducibility** — config + seed + commit hash are mandatory for every run. No hash = no finding.
4. **Atomic commits** — self-contained and non-breaking. Half-states don't exist; they're bugs.
5. **Plan large, implement small** — Opus for architecture, Sonnet subagents for >20 lines of code. No exceptions.
6. **Minimalism** — one source of truth per fact. Delete over deprecate. Complexity is compounding debt.
7. **Autonomy with trail** — agents act freely; every action is logged, every decision is traceable via commits + ClearML.
8. **Minimal overhead** — every tool justifies itself by saving more time than it costs. If a process requires a meeting, redesign it.
9. **Cross-pollination** — insights transfer across projects. Shared NFS exists so ideas bleed between directories.
10. **Ship or kill** — three months without progress means a deadline or a funeral. Be honest about which is which.

## Issue & Reporting Protocol

**Issue routing:**
- Infrastructure issues (bot, CI, tooling, governance) → `Fr0do/ouroboros`
- Research issues (training, experiments, paper, eval) → project repo (`Fr0do/s_cot`, `Fr0do/mmred`, etc.)
- Rule: file the issue where the code lives.

**Notion sync:**
- Collaborator-facing updates → Notion Research Log: findings, milestones, experiment results, paper status
- Implementation details → GitHub issues: bugs, config changes, pipeline tasks
- Rule: if a collaborator would care about it, it goes to Notion. If only you need to track it, it's a GitHub issue.

**Visibility:**
- Public website only reflects public repos. Private research stays invisible until the repo goes public.
