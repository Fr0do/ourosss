# OUROBOROS

> The serpent eats its own tail: each project feeds the next, and the meta-process improves itself.

A research operating protocol for automating the CS research paper pipeline from idea to publication.

---

## Pipeline

### Stage 0 · Ideation
Hypothesis + related work scan + feasibility check before any code.
Tools: `/deep-research`, `gh issue create`, Notion for eval framing.
Rule: no repo before a written hypothesis. An issue is the first artifact.

### Stage 1 · Bootstrap
Create LaTeX repo, GitHub repo, register in Notion.
Tools: `/deep-research` (scaffolds LaTeX + refs), `/feature` (files infra issues).
Rule: repo + paper skeleton exist before the first training run.

### Stage 2 · Infrastructure
Training script, data pipeline, eval harness wired up and smoke-tested.
Tools: **codex** subagent via Swarm for implementation (>20 lines); fall back to **gemini** for long-context / multimodal / paper edits or when codex is rate-limited. See `CLAUDE.md` for the routing matrix and `RALPH.md` for backlog-style ralph mode.
Rule: config + seed + commit hash logged from run zero. No reproducibility debt allowed to accumulate.

### Stage 3 · Training & Debugging
Interactive loop: launch → monitor → inspect → adjust.
Tools: bot `/run`, `/logs`, `/gpu`, `/disk`, `/ckpt`, `/completions`; `/check-training`, `/check-gpu` skills.
Rule: every run is tracked. Checkpoint every N steps. Kill diverging runs immediately.

### Stage 4 · Results & Analysis
Ablations, baselines, figures, tables — nothing goes in the paper without a tracked run.
Tools: bot `/metrics`, `/eval`, `/completions`, `/research-log`, Notion eval DB.
Rule: baselines run from the same config/env. Figures generated by script, not by hand.

### Stage 5 · Paper
Write, iterate, get feedback — parallel to late training.
Tools: `/deep-research` (related work refresh), **gemini** Swarm agent (`effort=detailed`) for long-context paper edits, **codex** for prose drafts and rewrites.
Rule: one source of truth — the LaTeX repo. No stale shadow copies.

### Stage 6 · Release
Camera-ready, artifact packaging, code release, portfolio update.
Tools: `gh release create`, `/feature` (release checklist issue), bot `/sync`.
Rule: artifact + paper + code released atomically. A result without reproducible code is an anecdote.

## Operational Rules

1. **Loop closes** — every artifact must improve the next iteration. Dead-end infra gets deleted.
2. **Issues before code** — create a GitHub issue before any feature. `fixes #N` in every commit.
3. **Reproducibility** — config + seed + commit hash are mandatory for every run.
4. **Atomic commits** — self-contained and non-breaking. Half-states don't exist; they're bugs.
5. **Plan large, implement small** — Opus is the only Anthropic model in the loop (orchestrator/planner/reviewer); codex via Swarm is the default executor, gemini is the fallback / long-context lane. No Sonnet/Haiku subagents.
6. **Minimalism** — one source of truth per fact. Delete over deprecate.
7. **Autonomy with trail** — agents act freely; every action is logged, every decision is traceable.
8. **Minimal overhead** — every tool justifies itself by saving more time than it costs.
9. **Cross-pollination** — insights transfer across projects.
10. **Ship or kill** — three months without progress means a deadline or a funeral.
